מעבדה

חומר מסכם על תשתית טיפוסית במעבדת Windows: שרתים, דומיין, רשת וקלאסטר כרעיון — בלי לייחס ציוד או גרסאות שלא תועדו כאן. ללינוקס יש לשונית נפרדת.

מבט כללי

מעבדה כזו משמשת לחקות חלק מסביבת ארגון: זהות (Active Directory), שירותי רשת בסיסיים, ותחנות קצה. רוב הרכיבים מרוצים כמכונות וירטואליות כדי לאפשר צילום מצב (snapshot) לפני שינוי והחזרה אחורה.

שרתים — מה זה אומר במעבדה

שרת בתכליתו הוא מחשב (פיזי או וירטואלי) שמריץ שירותים קבועים לרשת: אימות, שמות, הקצאת כתובות, קבצים וכו'. במעבדה ביתית נפוץ לרכז כמה תפקידים על מעט מכונות; בארגון אמיתי מפורקים לפי רישוי, ביצועים וזמינות.

  • מכונה וירטואלית — קובץ דיסק + הגדרות חומרה מדומה; רץ מעל Hyper-V, VMware Workstation/Fusion, או מארח עם KVM. מאפשר כמה “שרתים” על מחשב אחד.
  • תפקידי Windows Server שמופיעים בתרחיש AD: בקר דומיין (DC), לעיתים אותו שרת גם מחזיק DNS ו-DHCP; אפשר להוסיף שרת קבצים, שרת יישומים וכו' לפי מה שנלמד.
  • יותר משרת אחד — בתרגול זמינות אפשר להקים שני DC לאותו דומיין; כל DC מחזיק עותק של AD ומסנכרן עם השותף.

קלאסטר — עקרון (Windows ועולם מיכלים)

קלאסטר הוא קבוצת מחשבים (צמתים) שמתואמת כך שאותו שירות או עומס יכול לעבור ביניהם או לרוץ במקביל — כדי לשפר זמינות או להרחיב קיבולת.

  • Windows Failover Clustering — טכנולוגיה של Microsoft לזוג או יותר שרתי Windows שחולקים משאבים (למשל דיסק משותף או תצורת SMB). נדרש quorum (מנגנון הצבעה) כדי שהקלאסטר יחליט מי “בעלים” על המשאבים וימנע split-brain. שימושים מתועדים: למשל זמינות ל-SQL Server, או קבוצת מארחי Hyper-V.
  • קלאסטר מול מעבדת AD בלבד — AD עם כמה DC הוא כבר רפליקציה וזמינות בשכבת הזהות; Failover Clustering הוא שכבה נפרדת לשירותים ספציפיים שמוגדרים כמשאב מקובצ.
  • Kubernetes (בדרך כלל על לינוקס) — מערכת לאורכסטרציה של מיכלים על מספר מארחים: מתאימה לפריסת מיקרו-שירותים, לא לחלופה של Active Directory. נלמדת בנפרד מתשתית דומיין Windows.

הקמת סביבת עבודה (דומיין)

תרחיש בסיס שמופיע במעבדות מסוג זה:

  • Domain Controller (DC) — מחזיק את Active Directory, מנפיק Kerberos, משרת LDAP, ומנהל מדיניות.
  • קליינטים — Windows 10/11 (או שרת נוסף) מצורפים לדומיין כדי לבדוק כניסה, מדיניות קבוצתית, ושיתוף משאבים.

DNS ו־DHCP

  • DNS — מתרגם שמות לכתובות IP; בדומיין Active Directory אזורי DNS מסונכרנים בדרך כלל עם שם הדומיין והרשומות של שירותי Microsoft.
  • DHCP — מקצה לתחנות כתובת, מסכת רשת, שרת DNS ושער ברירת מחדל; מקל על הרחבת המעבדה בלי קונפיג ידני לכל קליינט.

תקשורת וחומת אש

  • בדיקות גילוי (למשל ping, שיתוף קבצים, RPC ל-AD) דורשות שכל הרכיבים באותה רשת לוגית או עם ניתוב מתאים.
  • Windows Defender Firewall — חוסם כברירת מחדל חלק מהתעבורה; במעבדה לעיתים מרככים כללים זמנית לצורך debug, ואז מחמירים שוב.

סקריפטים ואוטומציה

  • יצירת משתמשים ב-AD — אוטומציה ב-PowerShell (למשל New-ADUser) לעומת יצירה ידנית ב-ADUC.
  • יצירת OU — סקריפטים שמייצרים מבנה יחידות ארגוניות באופן חוזר, כדי לשמור על אחידות בין סביבות.

לינוקס

עקרונות, עמדת לינוקס לצד מעבדת Windows, ובלי לייחס מה בדיוק מותקן אצלך.

עמדת לינוקס — מה המערכת ואיך זה ליד Windows

לינוקס הוא מערכת הפעלה ממשפחת Unix: הליבה (Linux kernel) מנהלת חומרה ותהליכים; מעליה רצות תוכניות ושירותים. “הפצה” (למשל Debian, Ubuntu, RHEL, AlmaLinux, Fedora) היא שילוב של ליבה, ספריית חבילות, כלי ניהול ולעיתים ממשק גרפי אופציונלי.

בניגוד ל-Windows Server, רישוי ומודל תמיכה שונים, ודגש חזק על שורת פקודה, קבצי קונפיגורציה טקסטואליים ושירותים שמנוהלים לרוב עם systemd (בהפצות מודרניות רבות). גישה מרחוק טיפוסית היא SSH (מוצפן), לעומת RDP ב-Windows.

שימושים טיפוסיים במעבדות (כל אחד קיים בעולם האמיתי; לא מציין כאן מה בדיוק מותקן אצלך):

  • שרת אינטרנט — nginx או Apache HTTP Server.
  • DNS — למשל BIND (named); חלופה/תוספת ל-DNS של Windows כשלומדים רקורסיה והגדרות אזור.
  • DHCP — למשל isc-dhcp-server; עקרון ההקצאה דומה ל-Windows DHCP.
  • סקריפטים — Bash או שפות אחרות; אוטומציה דרך cron או systemd timers.
  • מיכלים — Docker או Podman להרצת יישומים מבודדים; Kubernetes לארגון מספר מארחים.

Active Directory הוא שירות שמיקרוסופט מממשת בעיקר ב-Windows; לינוקס יכול להצטרף לדומיין עם כלים כמו SSSD/Winbind (תלוי בתרחיש), או לעבוד כשרת עצמאי באותה רשת IP — חשוב להבדיל בין “אותה רשת פיזית” לבין “אותו דומיין מנוהל”.
